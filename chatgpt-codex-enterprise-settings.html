Part 01

ChatGPT Enterprise · Workspace Settings 진입 경로

ChatGPT Enterprise(및 Edu)에서 관리자가 워크스페이스 전체 설정을 다루는 곳은 화면 좌측 하단 프로필 아이콘 → Workspace settings 메뉴입니다. 이 한 화면에 멤버, seat(사용권), 권한, GPT 거버넌스, 앱(연결), 분석, ID 공급자(SSO/SCIM) 등 거의 모든 항목이 모여 있습니다.

진입 경로
ChatGPT 좌측 하단 프로필 아이콘 → Workspace settings

Owner와 Admin은 모든 항목을 변경할 수 있고, 일반 Member는 이 화면 자체를 열 수는 있어도 핵심 설정(권한·seat·지출·기능 토글)을 바꿀 수 없습니다. [OpenAI 공식]

Section 02 · ChatGPT

General — 일반 설정

Workspace settings → General

워크스페이스의 “겉모습”과 “기본 정책”을 정하는 탭입니다. 이름과 로고처럼 사용자가 매일 보는 요소, 그리고 조직 전체에 일괄 적용되는 AI 사용 정책 모달이 여기에서 관리됩니다.

02.1Workspace name & image

이름 옆 연필(편집) 아이콘을 눌러 워크스페이스 이름·이미지를 바꿉니다. 저장하면 모든 멤버 화면에 즉시 반영됩니다.

02.2Workspace instructions

조직 차원에서 ChatGPT가 “어떻게 답해야 하는지” 알려주는 안내문입니다. 전사 톤앤매너, 금지 표현, 공통 용어 등을 적어두면 모든 대화의 시스템 프롬프트에 추가됩니다.

02.3Workspace policy modal

조직 고유의 AI 사용 정책을 적어두면, 사용자가 처음 로그인할 때(그리고 정책을 갱신할 때마다) 강제 모달로 표시되어 동의를 받습니다. 컴플라이언스/감사 대비용으로 유용합니다.

02.4Workspace & organization identifiers

워크스페이스 ID와 OpenAI 조직 ID(org-id)를 확인할 수 있습니다. Compliance API, Microsoft Purview, Tenable 등 외부 거버넌스 도구를 붙일 때 이 값을 사용합니다.

참고
ChatGPT Workspace의 org-id와 API Platform 조직의 org-id는 원칙적으로 동일합니다. 둘이 다르게 보인다면 별도 조직이 만들어져 있을 가능성이 높으므로 Account Director에게 통합을 요청하세요. [SSO 개요]
Section 03 · ChatGPT

Members — 멤버 / Seat 관리

Workspace settings → Members

초대·삭제·역할 변경·seat 타입 변경이 모두 이 탭에서 일어납니다. Enterprise는 ChatGPT seat과 Codex seat을 자유롭게 섞어서 발급할 수 있고, 한 워크스페이스에 Codex-only seat만 둘 수도 있습니다.

03.1역할(Role) 4종

역할권한 요약
Owner결제·ID 관리·워크스페이스 구성 전체. 다른 Owner를 임명 가능.
Admin일상적인 사용자/그룹 관리. Owner 임명 불가.
Analytics Viewer 신규Member 권한 + Workspace Analytics 읽기 전용. AI 챔피언/팀 리더용.
MemberChatGPT/Codex 일반 사용. 관리 권한 없음.

→ 공식 역할 비교 표

03.2Seat 타입 2종

  • ChatGPT seat — 표준 ChatGPT 인터페이스(채팅·GPT·Apps·캔버스 등) 전체 사용.
  • Codex seat — Codex 사용 환경(App / CLI / IDE / Cloud / Code Review)만 사용. ChatGPT 채팅·메모리·프로젝트 접근 불가.
⚠ seat 변경 주의
ChatGPT seat을 Codex seat으로 바꾸면 해당 사용자의 채팅 기록·메모리·프로젝트(공유 프로젝트 포함)가 화면에서 사라집니다. 데이터가 삭제된 것은 아니며 ChatGPT seat으로 되돌리면 복구됩니다.

03.3주요 작업

  • 초대(Invite member) — 이메일 직접 입력 또는 CSV 업로드(email,role,seat type).
  • 대량 가져오기 — CSV로 한 번에 수백 명 등록 가능(예: user1@co.com,member,Codex).
  • 역할 변경 — Owner만 가능.
  • Seat 변경 — Owner만 가능.
  • Pending invites/Requests — 보류 중 초대·가입 요청을 승인/거절.
  • 멤버 제거 — ··· 메뉴 → Remove member.
  • 기본 seat 타입(Default seat type) 설정 — 이후 신규 가입자(SCIM 포함)는 이 값으로 발급.

→ 멤버/seat/역할 관리 가이드

Section 04 · ChatGPT

Groups — 그룹

Workspace settings → Groups

그룹은 RBAC(역할 기반 접근 제어)와 GPT/프로젝트 공유의 “단위”입니다. 워크스페이스 안에서 직접 만들거나, IdP의 디렉터리 그룹을 SCIM으로 자동 동기화할 수 있습니다.

  • 그룹 생성·삭제, 멤버 추가/제거
  • 그룹 멤버십 검토
  • SCIM 동기화 그룹은 이름 옆에 SCIM 배지가 표시되어 직접 관리 그룹과 구분
  • 같은 이름의 그룹이 이미 있으면 SCIM이 덮어쓰지 않고 별개 그룹으로 생성됨
RBAC 사용 시 필수
역할(권한)은 그룹에 부여합니다. 따라서 RBAC를 쓰려면 그룹이 반드시 필요합니다. 개인 사용자에게 직접 역할을 붙일 수는 없습니다. [SCIM FAQ]
Section 05 · ChatGPT

Permissions & Roles — 핵심 거버넌스 탭

Workspace settings → Permissions & Roles

Enterprise/Edu 관리자에게 가장 중요한 한 화면. 워크스페이스 전체에 적용되는 기본 권한, 그리고 그룹 단위로 더 엄격하거나 더 관대한 Custom Role을 만드는 곳입니다.

05.1Default permissions(기본 권한)

Custom Role이 없는 사용자에게 적용되는 워크스페이스 기본값입니다. 스크롤하며 토글로 기능별 ON/OFF를 결정합니다.

유의
Custom Role을 가진 사용자에게는 Default가 적용되지 않습니다. “전 직원에게 켜두고 싶은 기능”도 Custom Role 안에 다시 명시해야 합니다. [RBAC 가이드]

05.2Custom Roles — 토글 가능한 권한 카테고리

각 Custom Role 안에서 다음 항목을 개별 ON/OFF 할 수 있습니다(2026년 1월 기준).

  • Canvas — 캔버스 코드 실행 / 네트워크 접근 기본값
  • ChatGPT agent — 에이전트 모드(웹 자동 조작) 사용
  • Codex 사용 권한 — Codex 로컬/클라우드 액세스(아래 Codex 섹션 참조)
  • Apps(Connectors) — 앱 사용 허용 여부, 앱별 선택, action 단위 제어
  • GPTs — 생성·편집·공유 범위(invite-only / workspace / public), 서드파티 GPT 접근
  • Project creation — 프로젝트 생성 권한
  • Record (Memory/Notes) — 메모리/노트 기능
  • Search — 웹 검색, 에이전트 검색
  • Shared projects — 공유 프로젝트 참여
  • Link sharing — 대화 링크 공유 범위
  • Available models — 이 Role이 쓸 수 있는 모델 한정
  • Usage limits / Spend controls — 크레딧/메시지 한도(아래 참고)

여러 Custom Role을 동시에 가진 사용자는 합집합(가장 관대한 권한)으로 해석됩니다.

05.3Spend controls (Usage limits)

팀별/Role별로 크레딧 풀을 분배해, 특정 그룹이 예상치 못한 과다 사용을 일으키지 않도록 막을 수 있습니다. AI 챔피언 그룹에는 넉넉히, 일반 사용자 그룹에는 보수적으로 할당하는 식. [관련 문서]

05.4Early Access Features

정식 출시 전 신기능을 워크스페이스 단위로 켜고 끄는 영역. Owner만 토글 가능합니다.

Available Models — 모델 가시성

Workspace settings → Permissions & Roles 또는 별도 “Available Models” 영역

워크스페이스에서 어떤 모델을 노출할지 선택합니다. 예: GPT-5 계열, GPT-4 계열, o3/o4-mini, GPT-5-codex 등. Custom Role 안에서도 “이 Role이 쓸 수 있는 모델”을 별도 제한할 수 있어, 비싼 모델은 일부 그룹에만 열어두는 식의 통제가 가능합니다.

Section 07 · ChatGPT

GPTs — 거버넌스 / 소유권 / Action 도메인

Workspace settings → GPTs

조직 안에서 만들어진 모든 Custom GPT를 관리자 입장에서 통제하는 영역입니다. Enterprise는 Business 대비 훨씬 세밀한 제어가 가능합니다.

→ GPT access/governance 공식 가이드

07.1생성·편집·공유 권한

  • 누가 GPT를 만들거나 편집할 수 있는지(Role 단위로 결정)
  • 최대 공유 범위 — invite only / workspace only / broader
    • Enterprise: 워크스페이스 외부 공유를 완전히 차단 가능
    • Business: 사용자가 admin 승인 없이 공유 가능

07.2서드파티 GPT 접근

  • 모두 허용
  • Owner가 승인한 GPT만 허용
  • 전면 차단

07.3GPT 소유권 관리

  • 워크스페이스의 모든 GPT 일람
  • 소유자가 워크스페이스에서 제거되면 GPT는 자동으로 워크스페이스 Owner에게 이관되고 “unassigned”로 표시
  • 원래 작성자가 다시 추가되면 자동 복구. Owner는 언제든 다른 멤버에게 수동 재할당 가능
  • GPT 삭제(Deleting GPTs)도 이 화면에서 처리

07.4Action / Allowed domains

GPT가 외부 API를 호출하는 Action은 보안상 가장 위험할 수 있는 부분입니다. Owner는 허용 도메인 목록(Allowed domains)을 지정해, 그 외 도메인을 호출하는 Action은 실행되지 않도록 막을 수 있습니다. 부모 도메인을 등록하면 서브도메인도 함께 허용됩니다. 목록이 비어 있으면 모든 Action 차단입니다. [Action 도메인 설정]

07.5GPT 안에서 Apps 사용 허용

워크스페이스 안에서 만든 GPT가 Apps(Connectors)를 호출할 수 있는지 여부. 이 토글은 워크스페이스 단위 단일 설정이며, 서드파티 GPT에는 적용되지 않습니다.

Section 08 · ChatGPT

Apps (구 Connectors) — 외부 도구 연결

Workspace settings → Apps

2025년 12월 17일 OpenAI는 “Connectors”와 “Apps”를 Apps라는 단일 명칭으로 통합했습니다. Google Drive, SharePoint, Outlook, Slack, GitHub, Salesforce, Jira, Linear, Asana 등 외부 도구가 모두 여기서 관리됩니다.

→ Apps admin controls 공식 가이드

08.1워크스페이스 단위 설정

  • Enterprise/Edu는 모든 앱이 기본 OFF — Owner가 켜야 사용 가능
  • 앱별로 활성/비활성 토글
  • 활성화 시 admin은 어떤 Custom Role에게 노출할지 즉시 지정
  • 앱이 action 단위 제어를 지원하면 action별 검토·차단 가능

08.2Sync(인덱싱) 설정

  • Google Drive, SharePoint, GitHub 등은 사전 인덱싱(synced connector) 지원 — 응답 품질↑, 속도↑
  • 특정 Shared Drive/폴더만 허용, 특정 파일 형식 제외
  • 두 가지 셋업 방식:
    • Quick setup — 사용자가 자기 계정으로 인증
    • Admin-controlled access — 관리자가 중앙에서 더 세밀하게 제어

08.3RBAC for Apps

Permissions & Roles → Custom Role → Connected data 영역에서 Allow members to use apps 토글을 켜고 Select로 특정 앱만 노출합니다. 즉, “Marketing 그룹은 Slack/Drive만, Engineering은 GitHub/Linear까지” 식의 분리가 가능합니다. [Apps 관리]

08.4Custom MCP Apps (Developer mode)

  • OpenAI 미검증 — 신뢰하는 자체 앱만 등록할 것
  • Owner 또는 “developer mode 허용”된 사용자만 등록·게시 가능
  • Member 역할은 자체 등록 불가
  • RBAC로 누가 개발/테스트/사용할지 분리

08.5Microsoft 계열 연결 시 주의

Outlook, Teams, SharePoint 연결은 Microsoft Entra ID(구 Azure AD)에서 각 서비스마다 권한 부여가 선행되어야 합니다. Google Workspace 측도 Drive/Docs/Sheets/Slides 스코프 재인증이 필요한 경우가 있습니다.

Section 09 · ChatGPT

Workspace Analytics — 도입률·임팩트 측정

Workspace settings → Workspace analytics

“구매한 seat이 실제로 쓰이고 있는가?”에 답하는 영역입니다. 2025–2026년에 대대적으로 개편되어 Overview / Usage / Task Insights / Impact 4개 탭과 Analytics Viewer 신규 역할이 추가되었습니다.

09.1Overview 탭

  • 구매 seat · 활성화 seat · 활성화율
  • 주간 활성 사용자(WAU), 일일 활성 사용자(DAU)
  • 활성화 / 첫 사용 / 반복 사용 단계별 추세
  • 날짜·SCIM 그룹(부서) 필터

09.2Usage(사용량)

  • 총 메시지, 사용자당 메시지, 모델별·도구별·연결앱별 분포
  • SCIM 그룹별 비교(10명 미만 그룹은 익명화 위해 표시 제외)
  • CSV 내보내기 — Excel에서 바로 열 수 있는 on-demand 리포트

09.3Export data — Excel용 CSV 다운로드

Workspace Analytics의 Export는 현재 CSV 형식만 지원합니다. Excel 파일(.xlsx)을 직접 받는 방식은 아니지만, CSV를 Excel/Sheets에서 열어 사용자·GPT·프로젝트·임팩트 설문 분석용으로 사용할 수 있습니다.

Codex Policies — 클라우드 관리 requirements.toml

chatgpt.com/codex/settings/policies

개발자 노트북에 디바이스 파일을 직접 뿌리지 않고도, 그룹별로 다른 Codex 로컬 제약을 적용할 수 있는 화면입니다. 여기서 만든 정책은 사용자가 ChatGPT로 Codex에 로그인하는 순간 App / CLI / IDE 모두에 자동 동기화됩니다.

14.1제어 가능한 항목

  • allowed_approval_policies — untrusted, on-request 등 승인 모드 허용 목록
  • allowed_sandbox_modes — read-only / workspace-write / danger-full-access 중 허용 목록
  • allowed_web_search_modes — disabled / cached / live
  • MCP 서버 허용 목록 — 어떤 MCP 서버만 허용할지 이름·identity로 매칭. 빈 배열이면 전 MCP 차단
  • Restrictive command rules — rm, git push 같은 위험 명령에 prompt/forbidden 적용
  • Browser Use, in-app browser, Computer Use 등 특정 사용 환경 비활성화

14.2예시 정책

allowed_web_search_modes = ["disabled", "cached"]
allowed_sandbox_modes    = ["workspace-write"]
allowed_approval_policies = ["on-request"]

[rules]
prefix_rules = [
  { pattern = [{ token = "rm" }], decision = "forbidden",
    justification = "Use git clean -fd instead." },
  { pattern = [{ token = "git" }, { any_of = ["push", "commit"] }],
    decision = "prompt",
    justification = "Require review before mutating remote history." },
]

14.3그룹 할당 & 폴백

  • 각 정책을 특정 사용자 그룹에 할당
  • 매칭되지 않는 사용자용 기본(default fallback) 정책 별도 지정
  • 여러 그룹 룰에 매칭되면 맨 처음 매칭된 룰만 적용 — 누락 필드를 다음 룰이 채우지 않음
  • 각 정책은 “해당 그룹용 완성형 프로파일”로 작성할 것

14.4Policy lookup — 누구에게 어떤 정책이 적용되는지 확인

이메일 또는 그룹으로 조회해 적용 정책을 즉시 확인할 수 있습니다.

Cloud-managed requirements 외에도 각 사용자 PC에 정책 파일을 직접 배포해 Codex 동작을 강제할 수 있습니다. 이 섹션의 핵심은 OS마다 적용 방식·경로·한계가 모두 다르다는 점이며, 잘못 설계하면 정책이 강제되지 않거나 사용자가 우회할 수 있습니다.

15.1적용 우선순위 (3단계)

같은 항목을 여러 곳에서 정의하면 위쪽이 아래쪽을 덮어씁니다.

  1. Cloud-managed requirements — ChatGPT 웹의 Codex Policies 페이지. 사용자가 ChatGPT 로그인 시 자동 동기화. OS와 무관하게 적용되며 사용자가 우회 불가. ✅ 권장 1순위
  2. OS 레벨 관리형 설정 — macOS는 MDM, Linux는 /etc/codex/. Windows에는 동등한 위치 없음
  3. 사용자 개인 설정 — ~/.codex/config.toml. 사용자가 자유롭게 수정

20.5Compliance API와 Platform Audit Log API를 구분하세요

헷갈리기 쉬운 지점 Compliance API는 ChatGPT 워크스페이스의 사용자 활동, Codex, 인증, 감사 로그를 다루고, Admin/Audit Log API는 API Platform 조직의 API 키 생성, 프로젝트 관리, 멤버 변경 같은 관리 이벤트를 다룹니다. 둘은 별도 시스템이므로 ChatGPT Enterprise와 API Platform을 함께 쓰면 둘 다 모니터링해야 합니다.

20.7제공되지 않는 지표

OpenAI는 의도적으로 다음을 제공하지 않습니다: “생성된 코드 라인 수”(잘못된 인센티브 야기), “제안 수락률”(거의 100%라 의미 없음), “코드 품질/성능 KPI”(맥락에 너무 의존). 자체 KPI는 PR 리뷰 데이터·메타데이터로 직접 계산하라는 입장입니다.

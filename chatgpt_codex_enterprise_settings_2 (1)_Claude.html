Part 01

ChatGPT Enterprise · Workspace Settings 진입 경로

ChatGPT Enterprise(및 Edu)에서 관리자가 워크스페이스 전체 설정을 다루는 곳은 화면 좌측 하단 프로필 아이콘 → Workspace settings 메뉴입니다. 이 한 화면에 멤버, 시트(seat), 권한, GPT 거버넌스, 앱(연결), 분석, ID 공급자(SSO/SCIM) 등 거의 모든 항목이 모여 있습니다.

진입 경로
ChatGPT 좌측 하단 프로필 아이콘 → Workspace settings

Owner와 Admin은 모든 항목을 변경할 수 있고, 일반 Member는 이 화면 자체를 열 수는 있어도 핵심 설정(권한·시트·지출·기능 토글)을 바꿀 수 없습니다. [OpenAI 공식]

Section 02 · ChatGPT

General — 일반 설정

Workspace settings → General

워크스페이스의 “겉모습”과 “기본 정책”을 정하는 탭입니다. 이름과 로고처럼 사용자가 매일 보는 요소, 그리고 조직 전체에 일괄 적용되는 AI 사용 정책 모달이 여기에서 관리됩니다.

02.1Workspace name & image

이름 옆 연필(편집) 아이콘을 눌러 워크스페이스 이름·이미지를 바꿉니다. 저장하면 모든 멤버 화면에 즉시 반영됩니다.

02.2Workspace instructions

조직 차원에서 ChatGPT가 “어떻게 답해야 하는지” 알려주는 안내문입니다. 전사 톤앤매너, 금지 표현, 공통 용어 등을 적어두면 모든 대화의 시스템 프롬프트에 추가됩니다.

02.3Workspace policy modal

조직 고유의 AI 사용 정책을 적어두면, 사용자가 처음 로그인할 때(그리고 정책을 갱신할 때마다) 강제 모달로 표시되어 동의를 받습니다. 컴플라이언스/감사 대비용으로 유용합니다.

02.4Workspace & organization identifiers

워크스페이스 ID와 OpenAI 조직 ID(org-id)를 확인할 수 있습니다. Compliance API, Microsoft Purview, Tenable 등 외부 거버넌스 도구를 붙일 때 이 값을 사용합니다.

참고
ChatGPT Workspace의 org-id와 API Platform 조직의 org-id는 원칙적으로 동일합니다. 둘이 다르게 보인다면 별도 조직이 만들어져 있을 가능성이 높으므로 Account Director에게 통합을 요청하세요. [SSO 개요]
Section 03 · ChatGPT

Members — 멤버 / Seat 관리

Workspace settings → Members

초대·삭제·역할 변경·Seat(시트) 타입 변경이 모두 이 탭에서 일어납니다. Enterprise는 ChatGPT seat과 Codex seat을 자유롭게 섞어서 발급할 수 있고, 한 워크스페이스에 Codex-only seat만 둘 수도 있습니다.

03.1역할(Role) 4종

역할권한 요약
Owner결제·ID 관리·워크스페이스 구성 전체. 다른 Owner를 임명 가능.
Admin일상적인 사용자/그룹 관리. Owner 임명 불가.
Analytics Viewer 신규Member 권한 + Workspace Analytics 읽기 전용. AI 챔피언/팀 리더용.
MemberChatGPT/Codex 일반 사용. 관리 권한 없음.

→ 공식 역할 비교 표

03.2Seat 타입 2종

  • ChatGPT seat — 표준 ChatGPT 인터페이스(채팅·GPT·Apps·캔버스 등) 전체 사용.
  • Codex seat — Codex 표면(App / CLI / IDE / Cloud / Code Review)만 사용. ChatGPT 채팅·메모리·프로젝트 접근 불가.
⚠ 시트 변경 주의
ChatGPT seat을 Codex seat으로 바꾸면 해당 사용자의 채팅 기록·메모리·프로젝트(공유 프로젝트 포함)가 화면에서 사라집니다. 데이터가 삭제된 것은 아니며 ChatGPT seat으로 되돌리면 복구됩니다.

03.3주요 작업

  • 초대(Invite member) — 이메일 직접 입력 또는 CSV 업로드(email,role,seat type).
  • 대량 가져오기 — CSV로 한 번에 수백 명 등록 가능(예: user1@co.com,member,Codex).
  • 역할 변경 — Owner만 가능.
  • Seat 변경 — Owner만 가능.
  • Pending invites/Requests — 보류 중 초대·가입 요청을 승인/거절.
  • 멤버 제거 — ··· 메뉴 → Remove member.
  • 기본 시트 타입(Default seat type) 설정 — 이후 신규 가입자(SCIM 포함)는 이 값으로 발급.

→ 멤버/시트/역할 관리 가이드

Section 04 · ChatGPT

Groups — 그룹

Workspace settings → Groups

그룹은 RBAC(역할 기반 접근 제어)와 GPT/프로젝트 공유의 “단위”입니다. 워크스페이스 안에서 직접 만들거나, IdP의 디렉터리 그룹을 SCIM으로 자동 동기화할 수 있습니다.

  • 그룹 생성·삭제, 멤버 추가/제거
  • 그룹 멤버십 검토
  • SCIM 동기화 그룹은 이름 옆에 SCIM 배지가 표시되어 직접 관리 그룹과 구분
  • 같은 이름의 그룹이 이미 있으면 SCIM이 덮어쓰지 않고 별개 그룹으로 생성됨
RBAC 사용 시 필수
역할(권한)은 그룹에 부여합니다. 따라서 RBAC를 쓰려면 그룹이 반드시 필요합니다. 개인 사용자에게 직접 역할을 붙일 수는 없습니다. [SCIM FAQ]
Section 05 · ChatGPT

Permissions & Roles — 핵심 거버넌스 탭

Workspace settings → Permissions & Roles

Enterprise/Edu 관리자에게 가장 중요한 한 화면. 워크스페이스 전체에 적용되는 기본 권한, 그리고 그룹 단위로 더 엄격하거나 더 관대한 Custom Role을 만드는 곳입니다.

05.1Default permissions(기본 권한)

Custom Role이 없는 사용자에게 적용되는 워크스페이스 기본값입니다. 스크롤하며 토글로 기능별 ON/OFF를 결정합니다.

유의
Custom Role을 가진 사용자에게는 Default가 적용되지 않습니다. “전 직원에게 켜두고 싶은 기능”도 Custom Role 안에 다시 명시해야 합니다. [RBAC 가이드]

05.2Custom Roles — 토글 가능한 권한 카테고리

각 Custom Role 안에서 다음 항목을 개별 ON/OFF 할 수 있습니다(2026년 1월 기준).

  • Canvas — 캔버스 코드 실행 / 네트워크 접근 기본값
  • ChatGPT agent — 에이전트 모드(웹 자동 조작) 사용
  • Codex 사용 권한 — Codex 로컬/클라우드 액세스(아래 Codex 섹션 참조)
  • Apps(Connectors) — 앱 사용 허용 여부, 앱별 선택, action 단위 제어
  • GPTs — 생성·편집·공유 범위(invite-only / workspace / public), 서드파티 GPT 접근
  • Project creation — 프로젝트 생성 권한
  • Record (Memory/Notes) — 메모리/노트 기능
  • Search — 웹 검색, 에이전트 검색
  • Shared projects — 공유 프로젝트 참여
  • Link sharing — 대화 링크 공유 범위
  • Available models — 이 Role이 쓸 수 있는 모델 한정
  • Usage limits / Spend controls — 크레딧/메시지 한도(아래 참고)

여러 Custom Role을 동시에 가진 사용자는 합집합(가장 관대한 권한)으로 해석됩니다.

05.3Spend controls (Usage limits)

팀별/Role별로 크레딧 풀을 분배해, 특정 그룹이 예상치 못한 과다 사용을 일으키지 않도록 막을 수 있습니다. AI 챔피언 그룹에는 넉넉히, 일반 사용자 그룹에는 보수적으로 할당하는 식. [관련 문서]

05.4Early Access Features

정식 출시 전 신기능을 워크스페이스 단위로 켜고 끄는 영역. Owner만 토글 가능합니다.

Section 06 · ChatGPT

Available Models — 모델 가시성

Workspace settings → Permissions & Roles 또는 별도 “Available Models” 영역

워크스페이스에서 어떤 모델을 노출할지 선택합니다. 예: GPT-5 계열, GPT-4 계열, o3/o4-mini, GPT-5-codex 등. Custom Role 안에서도 “이 Role이 쓸 수 있는 모델”을 별도 제한할 수 있어, 비싼 모델은 일부 그룹에만 열어두는 식의 통제가 가능합니다.

Legacy Model Access
Enterprise/Edu는 일정 기간 구버전 모델을 유지하는 옵션이 별도로 제공됩니다. [Enterprise 개요]
Section 07 · ChatGPT

GPTs — 거버넌스 / 소유권 / Action 도메인

Workspace settings → GPTs

조직 안에서 만들어진 모든 Custom GPT를 관리자 입장에서 통제하는 영역입니다. Enterprise는 Business 대비 훨씬 세밀한 제어가 가능합니다.

07.1생성·편집·공유 권한

  • 누가 GPT를 만들거나 편집할 수 있는지(Role 단위로 결정)
  • 최대 공유 범위 — invite only / workspace only / broader
    • Enterprise: 워크스페이스 외부 공유를 완전히 차단 가능
    • Business: 사용자가 admin 승인 없이 공유 가능

07.2서드파티 GPT 접근

  • 모두 허용
  • Owner가 승인한 GPT만 허용
  • 전면 차단

07.3GPT 소유권 관리

  • 워크스페이스의 모든 GPT 일람
  • 소유자가 워크스페이스에서 제거되면 GPT는 자동으로 워크스페이스 Owner에게 이관되고 “unassigned”로 마킹
  • 원래 작성자가 다시 추가되면 자동 복구. Owner는 언제든 다른 멤버에게 수동 재할당 가능
  • GPT 삭제(Deleting GPTs)도 이 화면에서 처리

07.4Action / Allowed domains

GPT가 외부 API를 호출하는 Action은 보안상 가장 위험할 수 있는 부분입니다. Owner는 허용 도메인 목록(Allowed domains)을 지정해, 그 외 도메인을 호출하는 Action은 실행되지 않도록 막을 수 있습니다. 부모 도메인을 등록하면 서브도메인도 함께 허용됩니다. 목록이 비어 있으면 모든 Action 차단입니다. [Action 도메인 설정]

07.5GPT 안에서 Apps 사용 허용

워크스페이스 안에서 만든 GPT가 Apps(Connectors)를 호출할 수 있는지 여부. 이 토글은 워크스페이스 단위 단일 설정이며, 서드파티 GPT에는 적용되지 않습니다.

Section 08 · ChatGPT

Apps (구 Connectors) — 외부 도구 연결

Workspace settings → Apps

2025년 12월 17일 OpenAI는 “Connectors”와 “Apps”를 Apps라는 단일 명칭으로 통합했습니다. Google Drive, SharePoint, Outlook, Slack, GitHub, Salesforce, Jira, Linear, Asana 등 외부 도구가 모두 여기서 관리됩니다.

08.1워크스페이스 단위 설정

  • Enterprise/Edu는 모든 앱이 기본 OFF — Owner가 켜야 사용 가능
  • 앱별로 활성/비활성 토글
  • 활성화 시 admin은 어떤 Custom Role에게 노출할지 즉시 지정
  • 앱이 action 단위 제어를 지원하면 action별 검토·차단 가능

08.2Sync(인덱싱) 설정

  • Google Drive, SharePoint, GitHub 등은 사전 인덱싱(synced connector) 지원 — 응답 품질↑, 속도↑
  • 특정 Shared Drive/폴더만 허용, 특정 파일 형식 제외
  • 두 가지 셋업 방식:
    • Quick setup — 사용자가 자기 계정으로 인증
    • Admin-controlled access — 관리자가 중앙에서 더 세밀하게 제어

08.3RBAC for Apps

Permissions & Roles → Custom Role → Connected data 영역에서 Allow members to use apps 토글을 켜고 Select로 특정 앱만 노출합니다. 즉, “Marketing 그룹은 Slack/Drive만, Engineering은 GitHub/Linear까지” 식의 분리가 가능합니다. [Apps 관리]

08.4Custom MCP Apps (Developer mode)

  • OpenAI 미검증 — 신뢰하는 자체 앱만 등록할 것
  • Owner 또는 “developer mode 허용”된 사용자만 등록·게시 가능
  • Member 역할은 자체 등록 불가
  • RBAC로 누가 개발/테스트/사용할지 분리

08.5Microsoft 계열 연결 시 주의

Outlook, Teams, SharePoint 연결은 Microsoft Entra ID(구 Azure AD)에서 각 서비스마다 권한 부여가 선행되어야 합니다. Google Workspace 측도 Drive/Docs/Sheets/Slides 스코프 재인증이 필요한 경우가 있습니다.

Section 09 · ChatGPT

Workspace Analytics — 채택률·임팩트 측정

Workspace settings → Workspace analytics

“돈을 내고 산 시트가 실제로 쓰이고 있는가?”에 답하는 영역입니다. 2025–2026년에 대대적으로 개편되어 Overview / Usage / Task Insights / Impact 4개 탭과 Analytics Viewer 신규 역할이 추가되었습니다.

09.1Overview 탭

  • 구매 시트 · 활성화 시트 · 활성화율
  • 주간 활성 사용자(WAU), 일일 활성 사용자(DAU)
  • 활성화 / 첫 사용 / 반복 사용 단계별 추세
  • 날짜·SCIM 그룹(부서) 필터

09.2Usage(사용량)

  • 총 메시지, 사용자당 메시지, 모델별·도구별·연결앱별 분포
  • SCIM 그룹별 비교(10명 미만 그룹은 익명화 위해 표시 제외)
  • CSV / JSON 내보내기 — 주별/월별 단위 (커스텀 기간은 아직 미지원)

09.3Task Insights 기본 ON

  • 대화를 “Task 카테고리 / 토픽”으로 자동 분류한 집계 결과만 보여줌
  • 개별 프롬프트·대화는 절대 노출되지 않음
  • Heat map 으로 Task × 부서(SCIM 그룹) 교차 분석
  • Owner/Admin은 Workspace settings → Usage Analytics에서 OFF 가능 (이전에 만들어진 집계 데이터는 즉시 삭제되지 않음)

09.4Impact Surveys

Part 02 · Codex Enterprise

Codex Enterprise — 어디서 어떻게 설정하나

Codex(Enterprise)는 ChatGPT Enterprise 워크스페이스 안에 “포함”되는 형태입니다. 따라서 모든 관리 진입점은 동일한 Workspace Settings이며, Codex 전용 화면은 그 안에 별도 섹션으로 존재합니다. Codex는 Local(App/CLI/IDE 확장) 과 Cloud(웹/iOS/Code Review/Slack/Linear) 두 표면으로 나뉘며, 둘은 독립적으로 켜고 끌 수 있습니다.

Codex가 제공하는 Enterprise 보안
  • 고객 데이터로 학습하지 않음
  • App / CLI / IDE는 Zero Data Retention — 코드는 개발자 환경 밖으로 나가지 않음
  • 거주지(Residency)·보존(Retention) 정책은 ChatGPT Enterprise와 동일하게 적용
  • 저장 시 AES-256, 전송 시 TLS 1.2+
  • Compliance API로 모든 사용 기록 감사 가능

11.1이 문서의 OS 표기 규칙

Codex Local은 OS마다 설치/샌드박스/설정 위치가 갈리는 반면, Cloud·정책·Code Review 등은 OS 무관입니다. 아래 규칙으로 표기했습니다:

  • 🍎 macOS / 🪟 Windows / 🐧 Linux 배너 — 그 아래 내용은 해당 OS에만 적용
  • OS 무관 라벨 — macOS/Windows/Linux 모두 동일하게 동작
  • Linux 사용자에 대한 별도 안내는 Section 12 안의 🐧 Linux 영역에 모아둠

⭐ Native sandbox vs WSL2 — 어느 걸 쓸까

기준Native Windows sandboxWSL2
성능✅ 빠름 (권장)OK
설정 난이도관리자 1회 셋업 필요WSL2 설치만 하면 됨
기업 환경 호환일부 MDM이 sandbox 셋업 차단대부분의 환경에서 동작
파일 시스템Windows 경로 그대로WSL은 /home/... 권장 (/mnt/c는 느림)
리포지토리 위치아무데나WSL 홈에 두는 게 좋음

OpenAI 공식 권장은 native Windows sandbox. 다만 회사 노트북에서 sandbox 셋업이 권한 문제로 막히면 WSL2로 폴백하라는 가이드가 있습니다.

⚠ WSL1은 더 이상 지원되지 않음 Codex 0.115부터 Linux 샌드박스가 bubblewrap으로 이동하면서 WSL1이 빠졌습니다. 구버전을 쓰던 팀은 WSL2로 업그레이드 필요.

WSL2 설치 절차 (간단)

# 1. PowerShell 관리자로 열고
wsl --install   # Ubuntu 자동 설치

# 2. WSL Ubuntu 셸에서
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash
nvm install 22
npm i -g @openai/codex

# 3. 작업 폴더는 반드시 WSL 홈 안에
mkdir -p ~/code && cd ~/code
git clone https://github.com/your/repo.git
cd repo
codex

설정 위치

  • 사용자 설정: C:\Users\<계정>\.codex\config.toml (또는 WSL2 사용 시 ~/.codex/config.toml in WSL)
  • 인증 토큰: Windows Credential Manager
  • 관리형 정책: %USERPROFILE%\.codex\requirements.toml — Section 15의 Windows 부분 참조

샌드박스 디렉터리 추가

Native sandbox가 특정 폴더를 못 읽을 때 한시적으로 허용:

codex sandbox allow "C:\Projects\external-lib"

Q. Linux에서 Codex 쓸 수 있나요?
네, 충분히 쓸 수 있습니다. 다만 OpenAI는 Codex 데스크톱 앱(Electron 형태)을 Linux 빌드로 제공하지 않습니다. 대신 Codex CLI(완전 네이티브 musl 바이너리)와 VS Code/Cursor/Windsurf IDE 확장이 Linux에서 1급 시민으로 동작하며, 실제로 OpenAI 내부 엔지니어와 다수 기업 사용자가 Linux + CLI 조합을 메인으로 씁니다. 원격 SSH 서버, 데이터센터 빌드 박스, WSL2 환경 모두 같은 흐름으로 동작합니다.

설치 (Ubuntu/Debian 기준)

curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs

# Codex CLI 설치

# 또는 GitHub Releases에서 직접 다운로드

샌드박스 — bubblewrap (Codex 0.115+)

  • Codex CLI는 Linux에서 bubblewrap(bwrap) 기반 샌드박스를 사용

  • 샌드박스 모드: read-only / workspace-write / danger-full-access — macOS와 동일한 옵션

  • 인증 후 토큰은 OS별로 다른 위치에 저장: macOS Keychain / Windows Credential Manager / Linux ~/.codex/auth.json

  • Codex Admin 그룹 — 정책/환경/거버넌스 운영자(소수)
  • Codex Admin 그룹은 SCIM으로 IdP에서 관리 → 멤버십 변경이 감사 로그에 남음

여러 Role을 동시에 가진 사용자는 가장 관대한(least restrictive) 권한이 적용됩니다.

Section 14 · Codex

Codex Policies — 클라우드 관리 requirements.toml

chatgpt.com/codex/settings/policies

개발자 노트북에 디바이스 파일을 직접 뿌리지 않고도, 그룹별로 다른 Codex 로컬 제약을 적용할 수 있는 화면입니다. 여기서 만든 정책은 사용자가 ChatGPT로 Codex에 로그인하는 순간 App / CLI / IDE 모두에 자동 동기화됩니다.

14.1제어 가능한 항목

  • allowed_approval_policies — untrusted, on-request 등 승인 모드 화이트리스트
  • allowed_sandbox_modes — read-only / workspace-write / danger-full-access 중 허용 목록
  • allowed_web_search_modes — disabled / cached / live
  • MCP 서버 allowlist — 어떤 MCP 서버만 허용할지 이름·identity로 매칭. 빈 배열이면 전 MCP 차단
  • Restrictive command rules — rm, git push 같은 위험 명령에 prompt/forbidden 적용
  • Browser Use, in-app browser, Computer Use 등 특정 표면 비활성화

14.2예시 정책

allowed_web_search_modes = ["disabled", "cached"]
allowed_sandbox_modes    = ["workspace-write"]
allowed_approval_policies = ["on-request"]

[rules]
prefix_rules = [
  { pattern = [{ token = "rm" }], decision = "forbidden",
    justification = "Use git clean -fd instead." },
  { pattern = [{ token = "git" }, { any_of = ["push", "commit"] }],
    decision = "prompt",
    justification = "Require review before mutating remote history." },
]

14.3그룹 할당 & 폴백

  • 각 정책을 특정 사용자 그룹에 할당
  • 매칭되지 않는 사용자용 기본(default fallback) 정책 별도 지정
  • 여러 그룹 룰에 매칭되면 맨 처음 매칭된 룰만 적용 — 누락 필드를 다음 룰이 채우지 않음
  • 각 정책은 “해당 그룹용 완성형 프로파일”로 작성할 것

14.4Policy lookup — 누구에게 어떤 정책이 적용되는지 확인

이메일 또는 그룹으로 조회해 적용 정책을 즉시 확인할 수 있습니다.

Section 15 · Codex

Managed Configuration — OS별 완전 가이드

Cloud-managed requirements 외에도 각 사용자 PC에 정책 파일을 직접 배포해 Codex 동작을 강제할 수 있습니다. 이 섹션의 핵심은 OS마다 적용 방식·경로·한계가 모두 다르다는 점이며, 잘못 설계하면 정책이 강제되지 않거나 사용자가 우회할 수 있습니다.

15.1적용 우선순위 (3단계)

같은 항목을 여러 곳에서 정의하면 위쪽이 아래쪽을 덮어씁니다.

  1. Cloud-managed requirements — ChatGPT 웹의 Codex Policies 페이지. 사용자가 ChatGPT 로그인 시 자동 동기화. OS와 무관하게 적용되며 사용자가 우회 불가. ✅ 권장 1순위
  2. OS 레벨 관리형 설정 — macOS는 MDM, Linux는 /etc/codex/. Windows에는 동등한 위치 없음
  3. 사용자 개인 설정 — ~/.codex/config.toml. 사용자가 자유롭게 수정

15.2Requirements vs Managed defaults — 두 종류의 정책

구분파일 이름의미사용자 변경
Requirements (요구사항)requirements.toml강제 잠금. 위반 시 동작 자체가 차단됨❌ 불가
Managed defaults (관리형 기본값)managed_config.toml시작 기본값. 안전한 기본 상태로 시작⚠️ 세션 중 변경 가능 (다음 실행 시 다시 기본값 복원)

15.3OS별 지원 매트릭스 — 한눈에 보기

항목🍎 macOS🪟 Windows🐧 Linux
시스템 전역 설정 경로 /etc/codex/ 없음 /etc/codex/
사용자 홈 설정 경로 ~/.codex/ %USERPROFILE%\.codex\ ~/.codex/
OS 레벨 정책 강제 ✅ MDM (Jamf/Kandji/Mosyle/Fleet) ⚠️ 부분적 — Intune/GPO + ACL 조합 필요 ✅ /etc 권한 + 자동화 도구 (Ansible 등)
OpenAI 공식 MDM 키 com.openai.codex 도메인 ❌ ❌
실질적 강제 수단 MDM 프로파일 Cloud-managed requirements 필수 /etc 파일 + chmod 644 root

15.4공통 정책 예시

아래 TOML은 OS와 무관하게 동일하게 작성합니다. 차이는 “어디에 두느냐”뿐입니다.

# 보수적인 기본값
approval_policy = "on-request"
sandbox_mode    = "workspace-write"

[sandbox_workspace_write]
network_access = false   # 명시적으로 허용하지 않는 한 네트워크 OFF

[otel]
environment      = "prod"
exporter         = "otlp-http"
log_user_prompt  = false  # 프롬프트 본문 저장 금지(기본)
🍎macOS — MDM 기반 배포 가장 강력한 OS 레벨 강제

파일 위치

  • 관리형 기본값: /etc/codex/managed_config.toml
  • 요구사항(잠금): /etc/codex/requirements.toml
  • MDM 프로파일 도메인: com.openai.codex
  • MDM 키: config_toml_base64 (기본값) / requirements_toml_base64 (잠금)

MDM 워크플로 (Jamf Pro / Kandji / Fleet / Mosyle 공통)

  1. 위 TOML을 작성
  2. base64로 인코딩 (줄바꿈 없는 단일 라인으로): 
    cat requirements.toml | base64 | tr -d '\n' > requirements.b64
  3. MDM 프로파일에 다음 페이로드 추가: 
    <dict>
  <key>PayloadType</key>
  <string>com.openai.codex</string>
  <key>requirements_toml_base64</key>
  <string>[base64 인코딩 문자열]</string>
</dict>
  4. 대상 디바이스에 푸시 → 사용자에게 Codex 재시작 요청
  5. Codex 시작 로그(config summary)에 (managed) 표시 확인
  6. 정책 변경 시 페이로드만 갱신하면 다음 실행 때 자동 재적용

검증 명령

# 적용된 관리형 prefs 확인
defaults read com.openai.codex
# Codex가 인식한 정책 확인 (codex 실행 시 표시)
codex --print-config | grep managed
🪟Windows — 한계와 우회 전략 Cloud-managed requirements와 반드시 병행
⚠ Windows의 본질적 한계 OpenAI는 Windows용 시스템 전역 설정 경로(/etc/codex 등가물)와 전용 MDM 도메인을 현재 제공하지 않습니다. 설정 파일이 사용자 홈 폴더(%USERPROFILE%\.codex\)에만 존재하므로, 권한 있는 사용자는 파일을 삭제·수정할 수 있습니다. 따라서 실질적인 강제는 Cloud-managed requirements (Section 14)에서 해야 하며, 로컬 파일 배포는 “표준화 + 기본값 통일” 목적으로만 사용합니다.

파일 위치

  • 관리형 기본값: C:\Users\<계정>\.codex\managed_config.toml
  • 요구사항: C:\Users\<계정>\.codex\requirements.toml
  • 환경변수로 위치 변경 가능: CODEX_HOME=D:\codex-config

방법 ① Microsoft Intune (권장)

  1. Intune Admin Center → Devices → Configuration profiles → Create profile
  2. Platform: Windows 10 and later / Profile type: Templates → Custom
  3. OMA-URI로는 직접 파일 배포가 어려우므로 “Win32 app” 또는 “PowerShell scripts” 방식 사용
  4. 아래 PowerShell 스크립트를 “User context”로 실행하도록 배포:
$codexDir = Join-Path $env:USERPROFILE ".codex"
New-Item -ItemType Directory -Force -Path $codexDir | Out-Null

$reqContent = @'
approval_policy = "on-request"
sandbox_mode    = "workspace-write"

[sandbox_workspace_write]
network_access = false

[otel]
environment      = "prod"
exporter         = "otlp-http"
log_user_prompt  = false
'@

$reqPath = Join-Path $codexDir "requirements.toml"
Set-Content -Path $reqPath -Value $reqContent -Encoding UTF8 -Force

# 사용자에게 읽기만 허용, 시스템/관리자만 쓰기 가능
icacls $reqPath /inheritance:r `
  /grant:r "$env:USERNAME:(R)" `
  /grant:r "Administrators:(F)" `
  /grant:r "SYSTEM:(F)" | Out-Null

Write-Host "Codex requirements deployed to $reqPath"

방법 ② Group Policy (GPO) — 도메인 환경

  1. GPO → User Configuration → Preferences → Windows Settings → Files
  2. Source: 사내 SMB 공유 (\\fileserver\codex\requirements.toml)
  3. Destination: %USERPROFILE%\.codex\requirements.toml
  4. Action: Replace (매 로그온마다 갱신)
  5. 또는 Logon script로 위 PowerShell 코드 실행

방법 ③ SCCM / 단순 로그온 스크립트

:: codex-policy.bat — 로그온 스크립트
@echo off
if not exist "%USERPROFILE%\.codex" mkdir "%USERPROFILE%\.codex"
copy /Y "\\fileserver\codex\requirements.toml" "%USERPROFILE%\.codex\requirements.toml"
icacls "%USERPROFILE%\.codex\requirements.toml" /inheritance:r /grant:r "%USERNAME%:(R)" /grant:r "Administrators:(F)" /grant:r "SYSTEM:(F)"

사용자에게 안내할 메시지 템플릿

안녕하세요. 사내 Codex 사용 정책이 자동 적용됩니다.
배포 위치: C:\Users\<본인계정>\.codex\requirements.toml

🐧Linux — 시스템 파일 + 자동화 도구 /etc 기반 클래식 모델

파일 위치

  • 관리형 기본값: /etc/codex/managed_config.toml
  • 요구사항: /etc/codex/requirements.toml
  • 소유자/권한: root:root + chmod 644 (사용자 읽기 가능, 쓰기 불가)

수동 배포

sudo mkdir -p /etc/codex
sudo tee /etc/codex/requirements.toml > /dev/null << 'EOF'
approval_policy = "on-request"
sandbox_mode    = "workspace-write"

[sandbox_workspace_write]
network_access = false
EOF
sudo chown root:root /etc/codex/requirements.toml
sudo chmod 644 /etc/codex/requirements.toml

Ansible 플레이북 예시

- name: Deploy Codex managed requirements
  hosts: developers
  become: yes
  tasks:
    - name: Ensure /etc/codex exists
      file:
        path: /etc/codex
        state: directory
        owner: root
        group: root
        mode: '0755'

    - name: Copy requirements.toml
      copy:
        src: files/codex/requirements.toml
        dest: /etc/codex/requirements.toml
        owner: root
        group: root
        mode: '0644'

검증 명령

ls -la /etc/codex/
codex --print-config | grep -E "managed|requirements"

15.5공통 권장 가드레일

기본값 권장
  • 대부분 사용자 → workspace-write + on-request 승인
  • danger-full-access는 통제된 컨테이너용으로만
  • network_access = false 유지가 기본. 콜렉터/허용 도메인은 명시적 허용리스트로
  • log_user_prompt = false 유지 — 정책상 필요할 때만 ON
  • 로컬 config.toml과 관리형 정책의 diff를 주기 점검
  • OS별 차이를 인지하고, Windows는 반드시 Cloud-managed requirements와 병행

15.6검증 — config summary 읽는 법

Codex CLI/App을 실행하면 시작 시 “config summary”가 표시됩니다. 각 항목 옆에 (managed) 또는 (requirement) 표시가 있으면 OS 레벨/클라우드 정책이 적용된 것입니다. 표시가 없는 항목은 사용자 개인 설정(~/.codex/config.toml)에서 온 값입니다.

SIEM/eDiscovery/DLP에 붙여 사용자 프롬프트 본문, 응답, 토큰 사용량까지 감사 가능. 별도 API 키 + read/delete/둘 다 스코프 신청 필요. 

curl -L -H "Authorization: Bearer YOUR_COMPLIANCE_API_KEY" \
  "https://api.chatgpt.com/v1/compliance/workspaces/WORKSPACE_ID/logs?event_type=CODEX_LOG&after=2026-03-01T00:00:00Z"

주요 엔드포인트: /logs · /codex_tasks · /codex_environments

권장 패턴
대시보드(일상 모니터링) + Analytics API(자동화 보고) + Compliance API(감사·조사)를 동시에 운영하는 것이 표준 패턴입니다. [Governance 문서]

20.4제공되지 않는 지표

OpenAI는 의도적으로 다음을 제공하지 않습니다: “생성된 코드 라인 수”(잘못된 인센티브 야기), “제안 수락률”(거의 100%라 의미 없음), “코드 품질/성능 KPI”(맥락에 너무 의존). 자체 KPI는 PR 리뷰 데이터·메타데이터로 직접 계산하라는 입장입니다.

롤아웃 체크리스트 — 한 장 요약

  1. Owner / Security owner / Analytics owner 임명
  2. 도메인 검증 + SSO/SCIM 설정 (잠금 사고 방지를 위해 SSO는 일단 Optional 유지)
  3. 기본 시트 타입 결정 + 신규 사용자 자동 시트 발급 정책
  4. Default permissions와 Custom Roles, Spend controls 설계
  5. GPT 거버넌스(공유 범위·서드파티·Action 도메인) 결정
  6. Apps 워크스페이스 단위 활성/비활성 + 그룹별 노출
  7. Workspace Analytics에 Analytics Viewer 부여, 임팩트 설문 설계
  8. IP Allowlist 적용(필요 시)
  9. Codex Local/Cloud 토글, Codex Admin 그룹, Policies(requirements.toml) 정의
  10. OS별 클라이언트 배포 전략 수립: macOS는 MDM 프로파일, Windows는 Intune/GPO + Cloud-managed requirements 병행, Linux는 /etc/codex/ + Ansible 표준화
  11. Code Review · Codex Security 활성화
  12. Compliance API 키 발급 + SIEM 연동, Analytics API 자동화 리포트 구축
  13. 샘플 사용자로 최종 검증 — 로그인, MFA, RBAC, 관리형 정책, 분석 가시성
